# Волна 0 — безопасность: что изменено и как задеплоить

> ⚠️ Часть изменений (`server/`, `firebase-rules.json`) **не выкатывается push'ем в main**.
> Бэкенд — это Yandex Cloud Functions, правила — Firebase Console. Порядок деплоя важен.

## Что изменено

### Клиент (деплоится автоматически с `main` через Pages)
- `docs/games/duel/rt-firebase.js` — при создании/входе в комнату пишем `uid`
  (= `auth.uid` анонимной сессии) в `host`/`guest`. Нужно для новых правил RTDB.
- `docs/duel.html` — кэш-бастинг `duel.min.js?v=secfix1` (чтобы пользователи
  получили новый клиент до публикации правил).
- **Безопасно под старыми правилами**: лишнее поле `uid` они пропускают.

### Firebase RTDB (`firebase-rules.json`) — публиковать ВРУЧНУЮ
- Запись в `meta`/`host`/`guest` теперь привязана к `auth.uid` владельца слота —
  посторонний больше **не может перезаписать/закрыть чужую комнату**.
- Запись `messages` разрешена **только участникам** комнаты (host/guest по uid) —
  закрыт спуфинг сообщений от чужого имени.
- `from` в сообщении обязан совпадать с deviceId одного из участников + лимит длины.
- Чтение по-прежнему `auth != null` (модель «секретный код комнаты = доступ») —
  осознанный компромисс, иначе ломается вход гостя (он читает комнату до входа).

### Бэкенд (`server/`) — деплоить ВРУЧНУЮ как YC Functions
- `submit.js`:
  - **Анти-имперсонация**: для аутентифицированных ник берётся из JWT (`nick`),
    присланный в теле игнорируется (раньше можно было подставить чужой ник).
  - **Баг `NaN`**: нечисловое `time` больше не проходит анти-чит (`Number.isFinite`).
  - **Идемпотентность**: повторная отправка того же `matchId` → `409`, а не `500`.
  - Валидация `score`/`maxScore` на конечность.
- CORS во всех функциях: `Access-Control-Allow-Origin` берётся из env `ALLOW_ORIGIN`
  (по умолчанию `https://avvacumrechevoi.github.io`), а не `*`.

## Порядок деплоя (строго)

1. **Клиент** — смержить ветку в `main` и запушить (Pages соберёт и выкатит).
   Подожди ~2–3 мин, обнови `duel.html` в браузере (Ctrl+Shift+R), проверь, что
   создание комнаты пишет `host/uid` (DevTools → Network/RTDB).
2. **Firebase правила** — Firebase Console → Realtime Database → Rules →
   вставить содержимое `firebase-rules.json` → Publish.
   *Только после шага 1*, иначе старый клиент (без `uid`) не сможет создавать комнаты.
3. **Бэкенд** — передеплоить функции `submit`, `auth-telegram`, `leaderboard`,
   `content-publish`, `content-fetch` (yc CLI или Console). Задать env `ALLOW_ORIGIN`
   на боевой домен. `submit`/`content-publish` зависят от `JWT_SECRET`/`ADMIN_PASSWORD`.

## Как проверить (после деплоя)

- **Правила**: с друг­ого аккаунта/устройства попробовать записать в `rooms/<чужой код>/meta`
  через Firebase REST — должно вернуть `permission denied`. Своя комната — создаётся/играется.
- **submit анти-имперсонация**: `curl` с чужим `nickname` + валидным JWT → в БД
  пишется ник из токена, не из тела.
- **submit NaN**: `curl` с `time:"abc"` → `400 unrealistic time` (раньше проходило).
- **dup matchId**: повторный `curl` с тем же `matchId` → `409`.
- **PvP** (главное): сыграть реальную партию вдвоём по ссылке-комнате после публикации
  правил — создание, вход, ходы, финал работают.

## Ещё НЕ сделано (следующий заход по безопасности)
- **Rate-limiting** в `submit` (по `ip_hash`) — нужен счётчик в YDB, отложено.
- **content-publish**: статический пароль → заменить на JWT с ролью admin + lockout.
- **Server-issued match nonce** — чтобы счёт нельзя было сфабриковать в принципе
  (сейчас клиент всё ещё сообщает score; смягчено привязкой ника к токену).
- **Бэкапы YDB/RTDB** (Волна 0.5 в ROADMAP_IDEAL.md).

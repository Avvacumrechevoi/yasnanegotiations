# Регистрация и авторизация Ясны · план для РФ-аудитории

> **Статус: отложено · вернёмся позже.**
> На старте оставляем **только Telegram-логин** как самый простой способ входа без регистрации (одна кнопка → Telegram Login Widget → готово; не требует ни email, ни OAuth-приложений в VK/Яндекс, ни оператора ПДн в первой итерации, потому что Telegram сам передаёт минимум данных по своей схеме).
> Полный план ниже — для возврата, когда расширим способы входа.
>
> Документ — спецификация продуктового флоу до начала кода.
> Цель: договориться, **что** закрыто регистрацией, **как** пользователь её проходит, и **где** поведение единое во всех разделах.
> Аудитория: только РФ на старте. Стек: Yandex Cloud + YDB + VK ID + Яндекс ID + email/password fallback.

## 0 · Почему не Firebase Auth

В первой редакции этого документа предлагался Firebase Auth. Для русской аудитории это **не подходит**:

1. **152-ФЗ.** Email — персональные данные граждан РФ; они должны храниться в России. Firebase хранит всё в Google-датацентрах за пределами РФ. Это формальное нарушение, риск штрафов до 18 млн ₽.
2. **Доступ.** Письма от `noreply@firebaseapp.com` часто уходят в спам у `mail.ru` и `yandex.ru`. CDN Google периодически тормозит из РФ.
3. **Платежи.** Платный tier нельзя оплатить с российской карты. Бесплатный лимит закончится с ростом аудитории.

Поэтому — собственный backend на Yandex Cloud + рос.-friendly OAuth-провайдеры.

## 1 · Зачем нужна авторизация

- Сквозная идентичность: пройденный курс и накопленные ✦ доступны на любом устройстве при том же логине
- PvP с настоящими никами вместо «Гость №3247»
- Публичный лидерборд «Топ недели»
- Личная коллекция Ясн (для будущего конструктора)
- Знаки магистра (ачивки), привязанные к аккаунту

## 2 · Что закрыто, а что нет

Принцип: **гостя пускаем максимально далеко.** Регистрация не давится — предлагается только там, где без неё нельзя.

### Полностью открыто гостю

| Раздел | Что доступно |
|---|---|
| `start.html` | Лендинг, описание метода |
| `index.html` (Звезда) | Просмотр 12-полочной звезды, переключение Ясн |
| `learn.html` (Курс) | Прохождение всех 8 остановок, прогресс в `localStorage` |
| `duel.html` соло | Партия с Тенью без ограничений, бусины в `localStorage` |
| `rating.html` | Чтение объяснения системы рейтинга |

### Soft-wall · мягко зовём войти

Фича работает, но **не сохраняется на сервер**, и в ключевых точках мы об этом напоминаем без давления:

- В финальном экране соло-партии — кнопка «Сохранить прогресс · войти» рядом с обычными «Перепройти / Закрыть»
- В шапке аватар «Гость» с подсказкой: «Войди — прогресс сохранится во всех браузерах»
- На странице рейтинга мелким шрифтом: «У гостя рейтинг сбрасывается при очистке кэша»

### Hard-wall · действие невозможно без логина

Эти кнопки/действия **открывают модалку или страницу входа** при попытке использовать:

| Действие | Где встречается | Почему закрыто |
|---|---|---|
| Создать комнату PvP | `duel.html` → «С другом» | соперник должен видеть никнейм, а не «Гость» |
| Войти в чужую комнату PvP | `duel.html?room=KASTA-XXXX` | то же самое |
| Попасть в Топ недели | `duel.html` хроника | публичный лидерборд требует имени |
| Сохранить свою Ясну | будущий конструктор | нужен `owner_uid` |
| Поделиться своей Ясной | будущий конструктор | публичная ссылка с автором |
| Удалить аккаунт | страница профиля | по определению |

### Подтверждение email — отдельный gate

Регистрация через email прошла, но email ещё не подтверждён. Делаем ограниченный доступ — полные права только после клика по ссылке в письме.

| Действие | Email не подтверждён | Подтверждён |
|---|---|---|
| Соло-партия с сохранением | ✅ | ✅ |
| Курс с сохранением | ✅ | ✅ |
| Создать PvP-комнату | ❌ Баннер «Подтверди email» | ✅ |
| Войти в чужую PvP-комнату | ❌ | ✅ |
| Попасть в Топ недели | ❌ | ✅ |
| Опубликовать Ясну | ❌ | ✅ |

При входе через **VK ID** или **Яндекс ID** этот gate пропускается — провайдер уже подтвердил email.

## 3 · Способы входа

### MVP — Фаза 1

Главные кнопки на странице входа сверху вниз:

1. **Войти через VK** — кнопка с логотипом VK
2. **Войти через Яндекс** — кнопка с логотипом Яндекса
3. ─── разделитель `или` ───
4. Email/пароль — обычная форма ниже

VK ID и Яндекс ID — главные. По данным РФ-рынка ~80% пользователей предпочитают вход через них (один клик вместо ввода email и пароля). Email/password — fallback для тех, у кого нет ни одного аккаунта или кто принципиально хочет email.

### Что ещё подключаем позже

- **Mail.ru OAuth** — фаза 4–5
- **Sber ID** — фаза 5+
- **Google / Apple** — не подключаем для MVP. Могут понадобиться, если расширимся за РФ

## 4 · Стек на бэкенде

### Хостинг и БД

- **Yandex Cloud Functions** — auth-API: `auth-register`, `auth-login`, `auth-verify-email`, `auth-reset-password`, `auth-oauth-vk`, `auth-oauth-yandex`
- **Yandex Database (YDB)** — основная БД пользователей и сессий. Уже подключена в проекте (там лежат `matches`)
- **Yandex Object Storage** — для аватаров пользователей (если будет загрузка)
- **Yandex Cloud Lockbox** — секреты: HMAC-ключ для JWT, пароль SMTP, client_secret для VK/Яндекс
- **API Gateway** — единый домен `api.yasna.ru/auth/*` поверх функций

### Email-провайдер для писем подтверждения и сброса

Варианты:
1. **Яндекс 360 / Yandex Mail SMTP** — бесплатно до тысячи писем в месяц, домены лежат в РФ. Подходит для старта.
2. **Unisender / SendPulse** — российские транзакционные SMTP-сервисы, до 1500 писем бесплатно/месяц. Лучше доставляемость в `mail.ru` и `yandex.ru`.
3. **Mailgun / SendGrid** — западные, к сожалению, имеют те же проблемы что Firebase. **Не используем.**

Решение: **Unisender API** — у них есть прямая интеграция и хорошая доставляемость в основные русские почтовики.

### Хранение паролей

- Хэш `bcrypt` (cost = 12) или `argon2id` через `@node-rs/argon2`
- Никогда не храним plain-text
- HMAC-секрет для JWT в Lockbox, ротация раз в год

### Сессии

Два варианта стейтлесс/стейтфул:

**Вариант A · JWT в HttpOnly + Secure cookie**
- Подпись HS256 с ключом из Lockbox
- TTL 7 дней, refresh при каждом запросе
- При логауте — добавляем `jti` в YDB blacklist
- Минус: blacklist == по сути state, теряем «stateless»

**Вариант B · Случайный токен в YDB-таблице `sessions`**
- При логине → вставляем `(token, uid, expires_at)`
- Каждый запрос → SELECT по token
- Логаут → DELETE
- Плюс: проще логика инвалидации (типа «выйти на всех устройствах»)
- Минус: SELECT на каждый защищённый запрос (но YDB справляется)

**Решение: Вариант B**, потому что:
- проще для инвалидации
- YDB сильно дешевле и быстрее, чем платить за Redis
- сразу даёт фичу «активные устройства» в профиле

### Структура YDB

```
users
  uid              Utf8 (PK)         — uuid v4
  email            Utf8              — нормализованный (lowercase, trim), индекс
  email_hash       Utf8              — SHA256, для поиска без раскрытия email
  email_verified   Bool
  pwd_hash         Utf8 NULL         — argon2id; NULL если только OAuth
  nickname         Utf8              — публичный, уникальный
  nickname_lower   Utf8              — для unique-индекса
  avatar_hue       Int32             — 0..360 для генерации цвета аватара
  avatar_url       Utf8 NULL         — если будет загрузка
  oauth_vk_id      Utf8 NULL         — id из VK ID
  oauth_yandex_id  Utf8 NULL         — id из Яндекс ID
  created_at       Timestamp
  last_seen_at     Timestamp
  is_deleted       Bool DEFAULT false

sessions
  token            Utf8 (PK)         — random 32 bytes base64url
  uid              Utf8              — FK
  device_label     Utf8              — User-Agent fingerprint
  created_at       Timestamp
  expires_at       Timestamp
  last_used_at     Timestamp

  token            Utf8 (PK)
  uid              Utf8

  token            Utf8 (PK)
  uid              Utf8

```

## 5 · Регистрация и оператор ПДн

Для соблюдения 152-ФЗ нам нужно:

- [ ] **Юрлицо или ИП** — регистратор приложения VK ID и Яндекс ID, оператор ПДн
- [ ] **Подача уведомления в РКН** как оператор ПДн — бесплатно, заполняется онлайн в системе РКН за 30 минут, рассмотрение до месяца
- [ ] **Политика конфиденциальности** на сайте — обязательно. Шаблон от РКН + наши специфичные пункты
- [ ] **Согласие пользователя** — чекбокс при регистрации «Я согласен с обработкой персональных данных» с ссылкой на политику. Без согласия — кнопка регистрации заблокирована
- [ ] **Право на удаление** — пользователь может через профиль попросить удалить аккаунт. Реализуем soft-delete с retention 30 дней (на случай ошибочного удаления)
- [ ] **Право на скачивание данных** — экспорт JSON-архива в профиле
- [ ] **Сертификат SSL/TLS** — обязательный для передачи ПДн по HTTPS

Юридический минимум — порядка 1 рабочей недели на оформление + рассмотрение РКН.

## 6 · Сквозная сессия между страницами

Каждая HTML-страница (`start`, `index`, `duel`, `learn`, `rating`, `auth`, `profile`) подключает один общий модуль `core/auth-store.js`. Этот модуль:

1. На старте проверяет sessionToken в `localStorage`
2. Если есть — `GET /auth/me` → получает текущего юзера или 401
3. Сохраняет в памяти и эмитит `yasna-auth-changed`
4. Слушает `storage`-события: вход в одной вкладке → автоматически залогинены во всех

Глобальное API:

```
YasnaAuth.getUser()                            → { uid, email, emailVerified, nickname, avatarHue } | null
YasnaAuth.signUp({email, password, nickname})  → Promise<user>
YasnaAuth.signIn({email, password})            → Promise<user>
YasnaAuth.signInVK()                           → window.location to OAuth
YasnaAuth.signInYandex()                       → window.location to OAuth
YasnaAuth.signOut()                            → Promise
YasnaAuth.requestPasswordReset(email)          → Promise
YasnaAuth.subscribe(callback)                  → unsubscribe()
YasnaAuth.requireAuth(reason)                  → откроет /auth.html?reason=...&back=<current>
```

## 8 · Страница `/auth.html`

Одна страница, четыре режима через query-параметр `mode`:

- `?mode=signin` (по умолчанию) — вход
- `?mode=signup` — регистрация
- `?mode=reset` — забыл пароль (ввод email)
- `?mode=resetPassword&token=...` — переход из письма-сброса
- `?mode=verifyEmail&token=...` — переход из письма-подтверждения
- `?mode=oauthCallback&provider=vk|yandex&code=...` — callback от OAuth-провайдера

Параметр `back=<url>` — куда вернуть после успеха (по умолчанию `start.html`).

## 9 · OAuth-флоу (VK ID и Яндекс ID)

Для каждого провайдера регистрируем приложение в их dev-консоли:
- **VK ID:** [id.vk.com/business](https://id.vk.com/business) — нужен ИП/ООО
- **Яндекс ID:** [oauth.yandex.ru](https://oauth.yandex.ru/client/new) — можно от физлица для теста, для прода нужен ИП/ООО

В обоих провайдерах указываем:
- Redirect URI: `https://api.yasna.ru/auth/oauth/<provider>/callback`
- Scopes: `email`, `profile` (минимум — нам нужен email и имя)

### Шаги при входе через VK / Яндекс

1. Пользователь жмёт «Войти через VK» → клиент делает `window.location = 'https://id.vk.com/auth?…&redirect_uri=...&state=<random>'`
2. Пользователь логинится у VK, разрешает выдать email и имя
3. VK редиректит на `https://yasna.ru/auth.html?mode=oauthCallback&provider=vk&code=ABC&state=<random>`
4. Клиент шлёт `POST /auth/oauth/vk/exchange { code, state }` на наш бэкенд
5. Cloud Function `auth-oauth-vk`:
   - меняет `code` на `access_token` через VK API
   - получает email и имя пользователя
   - ищет в YDB user с `oauth_vk_id` = id из VK
   - если нашёл → создаёт session, возвращает токен
   - если не нашёл → ищет по `email_hash`. Если есть пользователь с тем же email → линкует VK к существующему аккаунту. Если нет — создаёт нового user (с предзаполненным никнеймом из VK, но позволяет сменить позже)
6. Клиент сохраняет sessionToken и редиректит на `back` URL

Аналогично для Яндекс ID.

### Линкование аккаунтов

Если пользователь сначала зарегистрировался через email, а потом нажал «Войти через VK» при том же email — мы автоматически связываем оба способа в одном `users.row`. В профиле показывается «Привязан VK ✓».

## 10 · Email-флоу (fallback)

### Регистрация через email

1. Форма signup → `POST /auth/register { email, password, nickname }`
2. Cloud Function:
   - валидирует поля
   - проверяет `email_hash` и `nickname_lower` в YDB на уникальность
   - хэширует пароль argon2id
   - создаёт user (`email_verified = false`)
   - генерирует токен `email_verifications`, TTL 24 часа
   - шлёт через Unisender письмо с ссылкой `https://yasna.ru/auth.html?mode=verifyEmail&token=...`
   - возвращает sessionToken (пользователь уже залогинен, но `emailVerified=false`)
3. На клиенте — баннер «Письмо отправлено. Проверь ящик»

### Подтверждение email

1. Пользователь кликает ссылку из письма
2. Страница `auth.html?mode=verifyEmail&token=...`
3. Клиент шлёт `POST /auth/verify-email { token }`
4. Cloud Function: ищет токен, помечает `email_verified = true`, удаляет токен
5. Редирект на `start.html` с тостом «Email подтверждён ✓»

### Сброс пароля

1. Форма «Забыл пароль» → `POST /auth/request-reset { email }`
2. Cloud Function:
   - находит user по `email_hash`
   - если найден → создаёт `password_resets` token, TTL 1 час, шлёт письмо
   - если не найден → **возвращает success в любом случае** (не раскрываем, что email не зарегистрирован — защита от перебора)
3. Пользователь кликает ссылку → `auth.html?mode=resetPassword&token=...`
4. Форма с полем «Новый пароль» → `POST /auth/reset-password { token, password }`
5. Cloud Function: проверяет токен, обновляет `pwd_hash`, удаляет токен, инвалидирует **все** существующие сессии этого user (на всякий случай)

## 11 · Миграция гостевого прогресса

Сценарий: гость наиграл 426 ✦, прошёл 3 темы курса. Регистрируется. Что с прогрессом?

При первой успешной авторизации `core/auth-store.js`:
1. Читает `localStorage` на наличие гостевого прогресса
2. Если есть — показывает модалку:

> У тебя в этом браузере уже сохранено: **426 ✦, 3 темы, 12 партий.** Перенести в новый аккаунт?
> [Перенести] [Начать с нуля]

3. «Перенести» → `POST /me/progress/merge { local_progress: {...} }`. Cloud Function:
   - читает текущий progress в YDB
   - применяет правило: `busey = max(local, server)`, темы — объединение, pr оставляем серверный (на сервере он точнее)
   - возвращает обновлённый progress

4. После мерджа — очищаем `localStorage` (кроме настроек темы и т.п.)

Что если человек:
- наиграл 100 ✦ дома
- пошёл с телефона, тоже залогинен, выиграл партию (+50 ✦)
- вернулся домой с открытой вкладкой

Решение: **источник правды — сервер.** При каждом начислении ✦ клиент:
1. Локально показывает обновлённое значение (для мгновенного отклика)
2. Шлёт `POST /me/match-finish { match: {...} }` на бэкенд
3. Cloud Function в YDB: `UPDATE progress SET busey = busey + delta WHERE uid = $uid;` — атомарно
4. Возвращает актуальное значение, клиент синхронизируется

Конкурентные записи не теряются — YDB поддерживает транзакции.

## 13 · Безопасность

### На сервере

- **HTTPS обязателен** — все cookies с `Secure; HttpOnly; SameSite=Lax`
- **Rate limit** на функциях:
  - `/auth/login`: 10 попыток/час с одного IP
  - `/auth/register`: 5 попыток/час с одного IP
  - `/auth/request-reset`: 3 попытки/час с одного email
- **Защита от перебора паролей** — после 5 неудач блокируем аккаунт на 15 минут
- **Anti-cheat для бусин** — `/me/match-finish` проверяет: дельта ≤ 200 за матч, не более 50 матчей в час. При нарушении — флаг подозрения

### На клиенте

- **Не храним пароли в localStorage** — только sessionToken
- **CSRF-защита** — `SameSite=Lax` cookie + проверка `Origin` header в функциях
- **XSS** — все динамические строки escape'ятся, никакого `innerHTML` с user-данными

### Содержимое JWT/sessionToken

- Токен — 32 случайных байта в base64url, 43 символа
- Никаких данных в самом токене (нет JWT) — только ключ к строке в YDB `sessions`. Это значит **мгновенный logout** работает, blacklist не нужен.

## 15 · Открытые вопросы

- **Возрастной gate?** Если делать публичный продукт — нужен чек «мне 16+». Пока пишем в условиях
- **Анонимные сессии?** Не используем — гость живёт в localStorage, аккаунт создаётся явным действием
- **Verify email — обязателен или опционален для соло?** В MVP оставляем обязательным **только для PvP/лидерборда**. Соло и курс работают без verification
- **Что если регистрируется в начале партии?** Партия не должна прерываться. Открываем `auth.html` в новой вкладке; результат партии присваивается новому аккаунту после возврата

## 16 · Принципы UI-копи

- Никогда «зарегистрируйтесь, пожалуйста» — просто «Войти / Зарегистрироваться»
- Никогда «без аккаунта потеряете всё» — это давление, неприятно
- Объяснять выгоду в одной строке: «Прогресс будет везде, где войдёшь»
- На ошибках использовать «попробуй ещё раз», а не «ошибка валидации»
- В подтверждении email звать по никнейму, не по email
- VK и Яндекс на кнопках — официальными иконками (брендбук обоих провайдеров)

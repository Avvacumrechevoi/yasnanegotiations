# Регистрация и авторизация Ясны · продуктовый план

> Документ — спецификация продуктового флоу до начала кода.
> Цель: договориться, **что** закрыто регистрацией, **как** пользователь её проходит, и **где** поведение единое во всех разделах.

## 1 · Зачем нужна авторизация

Сейчас весь прогресс хранится только в `localStorage` устройства. Это значит:
- сменил браузер — потерял бусины и пройденные темы
- очистил кэш — обнулился рейтинг и каста
- невозможно играть PvP с идентификацией соперника
- невозможно собрать общественный лидерборд («Топ недели»)
- невозможно сохранить собственные Ясны на сервере (когда появится конструктор)

Авторизация даёт сквозную идентичность — одно «я» во всех браузерах, во всех разделах, в PvP и в лидерборде.

## 2 · Что закрыто, а что нет

Принцип: **гостя пускаем максимально далеко.** Регистрацию предлагаем там, где без неё фича бессмысленна или неудобна.

### Полностью открыто гостю (никаких баннеров)

| Раздел | Что доступно |
|---|---|
| `start.html` | Лендинг, пресс-материалы |
| `index.html` (Звезда) | Просмотр 12-полочной звезды, переключение Ясн |
| `learn.html` (Курс) | Прохождение всех 8 остановок, прогресс в `localStorage` |
| `duel.html` соло-режим | Партия с Тенью без ограничений, бусины в `localStorage` |
| `rating.html` | Чтение объяснения системы рейтинга |

### Soft-wall · мягко зовём войти

Фича работает, но **не сохраняется на сервер**, и в нужный момент показываем приглашение:

- **После завершения соло-партии** в финальном экране одна кнопка вместо «Перепройти» — **«Сохранить прогресс · войти»**, и крестик «Закрыть» рядом. Не блокирует.
- **На главной** в шапке аватар «Гость» с подсказкой при наведении: «Войди — прогресс сохранится во всех браузерах».
- **На страница рейтинга** — в Hero под `Старт 1200` поясняем мелким шрифтом: «У гостя рейтинг сбрасывается при очистке кэша».

### Hard-wall · действие невозможно без логина

Эти кнопки/действия **открывают модалку входа** при попытке использовать:

| Действие | Где встречается | Почему закрыто |
|---|---|---|
| Создать комнату PvP | `duel.html` → «С другом» | соперник должен видеть никнейм, а не «Гость №3247» |
| Войти в чужую комнату PvP | `duel.html?room=KASTA-XXXX` | то же самое |
| Попасть в Топ недели | `duel.html` хроника | публичный лидерборд требует имени |
| Сохранить свою Ясну | будущий конструктор | нужен owner_id |
| Поделиться своей Ясной | конструктор | нужна публичная ссылка с автором |
| Удалить аккаунт | будущая страница профиля | сам по себе требует авторизации |

### «Доступно после email-подтверждения»

Регистрация прошла, но email ещё не подтверждён → даём ограниченный доступ. Полные права после клика по ссылке в письме.

| Действие | Без подтверждения email | После подтверждения |
|---|---|---|
| Соло-партия с сохранением | ✅ | ✅ |
| Учебный курс (прогресс на сервере) | ✅ | ✅ |
| Создать PvP-комнату | ❌ Баннер «Подтверди email» | ✅ |
| Войти в чужую PvP-комнату | ❌ | ✅ |
| Попасть в Топ недели | ❌ | ✅ |
| Опубликовать Ясну | ❌ | ✅ |

Цель — отсечь спам-аккаунты от публичных пространств (PvP, лидерборд, конструктор).

## 3 · Способы входа

**MVP (фаза 1):** Email + пароль. Этого достаточно — никаких внешних SSO.

**Фаза 2 (опционально):** Google OAuth и/или VK ID — одна кнопка вместо формы.

Apple Sign-In, Telegram, Yandex ID — пока **не** добавляем. Каждый провайдер — это отдельная поддержка.

### Что выбрать в форме регистрации

Минимально:
- **Email** (строка)
- **Пароль** — минимум 8 символов, требуем хотя бы одну цифру или спецсимвол
- **Имя в Ясне** (никнейм) — 3–20 символов, виден соперникам в PvP и в Топе недели. Уникальность проверяется при сабмите
- Чекбокс «Я согласен с условиями использования» — обязателен

Никаких «фамилия / возраст / город» — это бесполезно в первом релизе и режет конверсию.

### Что выбрать в форме входа

- Email
- Пароль
- Ссылка «Забыл пароль» — рядом с полем пароля

### Восстановление пароля

Стандартный Firebase-флоу:
1. Пользователь вводит email
2. Firebase шлёт письмо со ссылкой
3. Ссылка ведёт на `/auth.html?mode=resetPassword&oobCode=...`
4. Пользователь вводит новый пароль, попадает в систему

## 4 · Где живёт авторизация

Используем **Firebase Auth** в том же проекте `yasna-rt`, что уже подключён в `rt-firebase.js` для PvP. Это даёт:
- Email/Password провайдер из коробки
- Подтверждение email и сброс пароля без серверной логики
- Persistent session через `IndexedDB` — пользователь остаётся залогинен после закрытия вкладки
- Автоматическая синхронизация состояния между вкладками

Никаких новых backend-сервисов разворачивать не нужно. Все клиентские страницы могут подключить тот же `firebase-auth-compat.js`.

## 5 · Сквозная сессия между страницами

Каждая HTML-страница (`start`, `index`, `duel`, `learn`, `rating`, `auth`, будущий `profile`) подключает один общий модуль `core/auth-store.js`. Этот модуль:

1. Один раз инициализирует Firebase Auth с тем же `firebaseConfig`, что и rt-firebase
2. Подписывается на `onAuthStateChanged` и хранит текущего юзера в памяти
3. Эмитит событие `yasna-auth-changed` с `{ user, profile }` при любом изменении
4. Предоставляет API на `window.YasnaAuth`:

```
YasnaAuth.getUser()                 → { uid, email, emailVerified, nickname } | null
YasnaAuth.signUp({email, password, nickname})  → Promise<user>
YasnaAuth.signIn({email, password}) → Promise<user>
YasnaAuth.signOut()                 → Promise
YasnaAuth.resetPassword(email)      → Promise
YasnaAuth.subscribe(callback)       → unsubscribe()
YasnaAuth.requireAuth(reason)       → откроет /auth.html?reason=...&back=<current>
```

При входе или регистрации модуль создаёт/обновляет документ в RTDB:
```
/users/{uid}
  email          (приватно — не читается публично)
  nickname       (публично, проверяется на уникальность)
  avatarHue      (число, для генерации цвета аватара)
  createdAt
  lastSeenAt
  emailVerified  (mirror Firebase-поля)
```

И сводный профиль для лидерборда — `/leaderboard/{uid}` с агрегатами busey/rating, обновляется при сабмите матча.

## 6 · Шапка во всех страницах — единое поведение

В правом верхнем углу одно из двух состояний:

**Гость** — кнопка «Войти» (синяя, ведёт на `/auth.html?back=<current>`)

**Залогинен** — аватар (генерируется из avatarHue) + никнейм. Клик открывает выпадающее меню:
- Никнейм + email мелким шрифтом
- Прогресс: «Послушник VI · 591 ✦» (взято из профиля)
- Разделитель
- Профиль → `/profile.html`
- Настройки → `/profile.html#settings`
- Разделитель
- Выйти

Если email **не подтверждён** — рядом с никнеймом мелкая жёлтая точка и в меню баннер «Подтверди email» + ссылка «Отправить ещё раз».

## 7 · Страница `/auth.html`

Единая страница для входа, регистрации и сброса пароля. Состояние переключается через query-параметр `mode`:

- `?mode=signin` (по умолчанию) — вход
- `?mode=signup` — регистрация
- `?mode=reset` — забыл пароль (ввод email)
- `?mode=resetPassword&oobCode=...` — Firebase сюда ведёт из письма-сброса
- `?mode=verifyEmail&oobCode=...` — Firebase сюда ведёт из письма-подтверждения

Все формы — одна страница, один CSS, один JS-модуль. На входе у пользователя только то, что нужно.

Параметр `back=<url>` — куда вернуть после успеха (по умолчанию `start.html`).

### Структура страницы

```
[ ← Ясна ]                           (логотип-ссылка наверху слева)

           Войти в Ясну               (h1 в Cormorant Garamond)
           Сохрани прогресс во всех браузерах   (lead)

           ┌─────────────────────────┐
           │  Email                  │
           │  [_______________]      │
           │                         │
           │  Пароль                 │
           │  [_______________]      │
           │  (Забыл пароль?)        │
           │                         │
           │  [    Войти    ]        │
           │                         │
           │  ─────  или  ─────      │
           │                         │
           │  Ещё нет аккаунта?      │
           │  Зарегистрироваться →   │
           └─────────────────────────┘
```

В режиме регистрации добавляется поле «Имя в Ясне» и чекбокс согласия.

### Сообщения об ошибках

| Случай | Сообщение |
|---|---|
| Неверный email | «Email-адрес выглядит неправильно» |
| Слабый пароль | «Пароль должен быть не короче 8 знаков» |
| Email уже зарегистрирован | «Такой email уже есть. Может быть, ты хотел войти?» (с кнопкой переключения) |
| Никнейм занят | «Такое имя уже взято. Попробуй другое.» |
| Неверный пароль при входе | «Email и пароль не совпадают» (без раскрытия, что именно — на случай атак) |
| Слишком много попыток | «Слишком много попыток. Подожди 5 минут или сбрось пароль.» |

## 8 · Миграция гостевого прогресса

Сценарий: гость наиграл 426 ✦, прошёл 3 темы курса. Регистрируется. Что с прогрессом?

**Решение:** при регистрации показываем модалку:

> У тебя в этом браузере уже сохранено: **426 ✦, 3 темы, 12 партий.** Перенести в новый аккаунт?
> [Перенести] [Начать с нуля]

«Перенести» — мерджим `localStorage` → `/users/{uid}/progress` и очищаем гостевой стейт.
«Начать с нуля» — оставляем гостевой прогресс в localStorage (если разлогинятся, он там и будет), а в облачный аккаунт ничего не пишем.

При **повторном входе на новом устройстве** — наоборот, локальный прогресс пустой, всё подтягивается с сервера.

## 9 · Конфликт устройств

Что если человек:
- наиграл 100 ✦ дома
- пошёл играть с телефона, тоже залогинен
- получил там 50 ✦
- вернулся домой с открытой вкладкой

Решение: **источник правды — сервер.** При каждом начислении бусин клиент:
1. читает текущее значение с сервера через `transaction()`
2. прибавляет дельту
3. записывает обратно

Это атомарно — конкурентные записи не теряются.

Локальный кэш используется только для мгновенного отклика UI; «настоящее» значение — на сервере.

## 10 · Безопасность · что на чьей стороне

**Firebase Security Rules** для RTDB:
- `/users/{uid}` — читать и писать может только `auth.uid === uid`
- `/users/{uid}/nickname` — читается всеми (для отображения никнейма соперника)
- `/leaderboard/*` — читается всеми, пишется только Cloud Function (или с проверкой `pr_change ≤ K`)
- `/rooms/*` — текущая логика PvP остаётся

**Никогда не на клиенте:**
- начисление бусин нельзя писать прямо клиентом — иначе любой может прислать `busey: 999999`. **Решение для MVP:** временно доверяем клиенту + rate-limit на сервере. Позже — Cloud Function `submitMatch`, которая считает бусины из присланного матча и кладёт в профиль.

## 11 · Удаление аккаунта

Право пользователя — удалить себя.

Страница `/profile.html#delete` → кнопка «Удалить аккаунт» → подтверждение пароля → удаление:
1. Firebase Auth user
2. `/users/{uid}` — все поля кроме `nickname` (нужен для разрыва ссылок в Хронике), который анонимизируем как «Странник»
3. `/leaderboard/{uid}` — удалить
4. `/rooms` с этим пользователем — оставить (история PvP), заменив upper-id на `deleted-user`

GDPR-friendly: предлагаем перед удалением скачать архив (json) своего прогресса.

## 13 · Открытые вопросы

- **Возрастной gate?** Если делать публичный продукт — нужен чек «мне 16+». Пока не делаем, пишем в условиях.
- **Анонимные авторизации Firebase?** Можно вместо «гостя в localStorage» создавать анонимный uid. Плюс: всё уже на сервере. Минус: ещё один уровень состояния и lifecycle. Решение: **не используем** в MVP.
- **Verify email — обязательно или опционально?** В MVP оставляем обязательным **только для PvP/лидерборда**. Соло и курс работают без verification.
- **Что если пользователь регистрируется в начале партии?** Партия не должна прерываться. Открываем `auth.html` в новой вкладке/модалке поверх; результат партии присваивается новому аккаунту после возврата.
- **Реакция на смену устройства / выход одного юзера в нескольких вкладках?** Firebase Auth сам синхронизирует — выход в одной вкладке закрывает сессию во всех.

## 14 · Принципы для UI-копи

- Никогда не «зарегистрируйтесь, пожалуйста» — просто «Войти / Зарегистрироваться»
- Никогда не угрожать «без аккаунта потеряете всё» — это давление, неприятно
- Объяснять выгоду в одной строке: «Прогресс будет везде, где войдёшь»
- На ошибках использовать «попробуй ещё раз», а не «ошибка валидации»
- В подтверждении email звать по никнейму, не по email

# Дуэли — roadmap развития

Дата: 2026-05-07. Стартовое состояние: P0 (3 Race-режима через BroadcastChannel, ~760 строк кода в `docs/preview/games/duel/`).

## 0. Где мы сейчас (P0 — Foundation)

✅ **Готово:**
- Engine: `DuelTransport` + `DuelApp` + `Lobby` + `DuelRunner` + `ResultScreen`
- Реестр `window.YasnaDuels` с register/get/list/validate
- Параметризованный Race: 3 режима (Опорный Крест / Крест Управления / Крест Веры) × 3 Ясны (Суток/Года/Жизни) = **9 уникальных матчей**
- Стабильность: heartbeat (1.5s) + disconnect detection (6s) + matchId guard + sessionId фильтр + surrender + cleanup
- UX: pick-game → configure (yasna picker) → hosting/joining → countdown → game → result

⚠️ **Что не закрыто:**
- Только same-browser демо (BroadcastChannel) — нет реального online
- Нет профилей, истории, статистики
- Нет лидербордов, рейтингов
- Только Race-категория (Quiz, Mirror, Bidding, Tournament — на бумаге)
- Нет мобильной адаптации gameplay (Star рендерится как на десктопе)
- Нет туториала / первого match опыта

## 1. Принципы развития

| Принцип | Что значит на практике |
|---|---|
| **Изоляция** | Каждая фаза = отдельный подмодуль `docs/preview/games/duel/...`. Не правим `core/`, `app.js`, `tours/`, `lessons/`. |
| **Регрессии = недопустимо** | После каждой фазы — smoke-test: открыть Дуэль, сыграть Race-Cross, проверить result. Если что-то сломалось — откат. |
| **Free-tier first** | Любая инфраструктура должна влезать в free tier. Платные сервисы только когда продукт доказал ценность. |
| **Mobile-first для gameplay** | Дуэль чаще играют с телефона. Десктоп — для preview/hosting/админки. |
| **Контракт API стабилен** | `DuelTransport` интерфейс не меняется при миграции BroadcastChannel → WebRTC → WebSocket. Игры пишутся один раз. |
| **Прогрессивная сложность** | Игроку даём играть сразу (Race-Cross на Ясне Суток), потом открываем сложнее (Зодиак, Лента Мёбиуса). |
| **Без vendor-lock** | Cloudflare Workers / PartyKit можно заменить на свой сервер за день. PeerJS заменяется на собственный signaling за день. |

## 2. Шесть фаз развития

```
┌──────────────────────────────────────────────────────────────────────┐
│  P1  FOUNDATION    — UX, мобильник, ещё 3 режима, туториал         │  1 неделя
├──────────────────────────────────────────────────────────────────────┤
│  P2  ONLINE        — реальный WebRTC через PeerJS/Trystero          │  1.5 недели
├──────────────────────────────────────────────────────────────────────┤
│  P3  PERSISTENCE   — профили, история, локальная статистика         │  1 неделя
├──────────────────────────────────────────────────────────────────────┤
│  P4  COMPETITION   — leaderboard, рейтинг ELO, daily tournament     │  2 недели
├──────────────────────────────────────────────────────────────────────┤
│  P5  SOCIAL        — friends, challenge by name, replay             │  1.5 недели
├──────────────────────────────────────────────────────────────────────┤
│  P6  LONG-TAIL     — achievements, AI-практика, кастомизация         │  по желанию
└──────────────────────────────────────────────────────────────────────┘
```

Каждая фаза — точка остановки. Можно прекратить после любой и продукт остаётся пользоваемым. Между фазами — оценка пользы, поход на P+1 только если предыдущая «сработала».

## P1 — Foundation (~1 неделя)

**Цель:** довести существующий MVP до состояния, в котором не стыдно показать другу. Доделать гладкую UX и расширить каталог режимов до 6.

### P1.1 Mobile-first gameplay (½ дня)
- В Race на телефоне: бóльшие тапы, sticky status-bar, скрыть лишние подписи
- Звезда занимает 70% экрана, текст 30%
- Использовать `mob` props у `<Star>` который уже есть

### P1.2 Туториал «Первая дуэль» (½ дня)
- Single-player tutorial: бот «Андрей» делает 4 правильных полки за 30 сек
- Игрок учится механике без давления соперника
- Открывает Race-Cross на Ясне Суток после прохождения

### P1.3 Ещё 3 режима (~3 дня)
- **Quiz «Антиподы»** — 5 вопросов «антипод этой полки?», по очереди (~150 строк)
- **Mirror «Заполни Ясну»** — drag-and-drop 12 меток на 12 позиций (~250 строк)
- **Speed «Молния»** — 30 секунд, кликай как можно больше правильных полок Опорного Креста на разных Яснах (~120 строк)

### P1.4 Полировка лобби (½ дня)
- В каталоге игр — сортировка по сложности
- Pre-match preview: показать Ясну, на которой будет дуэль
- Visible "ping" к сопернику (loading dots)

### P1.5 Соединение с Уроками (¼ дня)
- После урока «Опорный Крест» (lesson-2) → CTA «Сыграть в дуэль»
- В лобби — пометка «требует урок 2» если не пройден

**Итого P1:** ~5 дней, +6 режимов в каталоге, mobile-ready, туториал.

---

## P2 — Online (1.5 недели)

**Цель:** заменить BroadcastChannel на реальный WebRTC, чтобы играть с друзьями через интернет.

### P2.1 Trystero P2P (~2 дня)
- Подключить [Trystero](https://github.com/dmotz/trystero) (~30 KB gzip) через CDN
- Реализовать `TrysteroTransport extends DuelTransport`
- Same API как у `BroadcastTransport` — игры менять не нужно
- Free torrent trackers как signaling

### P2.2 Fallback чейн (1 день)
- Сначала пробуем Trystero. Если signaling не работает (~5% случаев) — падаем в `firebase-trystero` бесплатный.
- Если совсем нет коннекта — показываем сообщение «попробуйте мобильный интернет / другой WiFi».

### P2.3 NAT-traversal (1 день)
- ~10% игроков не пройдут чистый P2P из-за симметричных NAT.
- Решение MVP: показать диагностику и совет «попросите соперника создать комнату вместо вас» (host обычно проходит легче).
- Решение Pro: TURN-сервер ($5/мес VPS Coturn). Делаем когда жалоб станет >5%.

### P2.4 Reconnection (1.5 дня)
- При обрыве соединения — попытка восстановления 10 секунд.
- Если матч был в активной фазе — сохраняем state локально, при reconnect синхронизируем.
- Если соединение не восстановилось за 10с — присуждаем победу оставшемуся.

### P2.5 Профиль гостя (½ дня)
- При первом входе в Дуэль: запрос nickname + аватар (12 emoji на выбор)
- Сохраняем в `localStorage['yasna_duel_profile']`
- Передаём в `hello`/`host-ack` — соперник видит твой ник в status-bar

### P2.6 Public lobby (1 день)
- Список открытых комнат: «Anna ищет соперника на Race-Cross / Ясна Суток»
- Реализуется через Trystero Room с типом `lobby` — все клиенты видят список
- Кликаешь — попадаешь в дуэль

**Итого P2:** ~8 дней, реальный online, профиль с ником, Public lobby.

**Точка решения:** если за неделю после релиза не больше 5 онлайн-дуэлей в день — стоп. Если больше 50 в день — катимся в P3.

## P3 — Persistence (~1 неделя)

**Цель:** сохранять историю матчей, личные рекорды, статистику.

### P3.1 Match history (1 день)
- После каждого матча запись в `localStorage['yasna_duel_history']`
- Поля: `{ id, date, mode, yasna, opponent, role, result, time, score }`
- Лимит 100 последних
- UI: «История» вкладка в лобби

### P3.2 Personal stats (½ дня)
- За каждый режим: побед/поражений, лучшее время, среднее время, win rate
- Тепловая карта: какие Ясны сильнее, какие хуже
- UI: «Статистика» страница

### P3.3 Best time per mode (½ дня)
- Таблица личных рекордов: «Race-Cross на Ясне Суток: 8.3s»
- Бейдж «новый рекорд» в Result Screen

### P3.4 Streak / serial (½ дня)
- Отдельный счётчик подряд побед в каждом режиме
- Бейджи: 5 подряд, 10 подряд, 20 подряд

### P3.5 Анти-чит для local stats (¼ дня)
- Если matchId не уникальный или time < 1s — не записываем
- Не считается реплеем

### P3.6 Export/import профиля (1 день)
- JSON экспорт всей истории/статистики
- Можно перенести с одного устройства на другое
- Optional: sync через Sign in with Google (будет в P4)

**Итого P3:** ~3 дня. Без сервера, всё local.

---

## P4 — Competition (2 недели)

**Цель:** ввести понятие «лучших игроков», публичные лидерборды, рейтинг ELO. Здесь впервые нужна серверная инфраструктура.

### P4.1 Cloudflare Worker setup (1 день)
- `wrangler init yasna-duel-server`
- Endpoints:
  - `POST /match` — записать результат матча
  - `GET /leaderboard?mode=race-cross&period=daily`
  - `GET /me` — мой рейтинг
- KV хранилище для leaderboard, D1 для матчей
- Free tier: 100k req/день, 1M KV ops/мес — на старте хватит

### P4.2 Anti-cheat baseline (1.5 дня)
- Server-side валидация:
  - matchId уникален
  - time не меньше теоретического минимума (например, для Race ≥ 4 кликов × 100ms = 400ms)
  - Оба игрока подтвердили исход
- Дисквалификация при подозрении (без бана, просто матч не засчитывается)

### P4.3 Sign in with Google (1.5 дня)
- OAuth через Workers Auth
- Заменяет guest-id на устойчивый `user-id`
- Прогресс из P3 мигрируется (offer на UI)

### P4.4 ELO rating (1 день)
- Стандартная формула. K-factor 32 для новичков, 16 для опытных.
- Per-mode рейтинги (отдельно Race-Cross, Race-Mngmt и т.д.)
- Глобальный = средний по режимам

### P4.5 Leaderboard UI (1 день)
- Top-10 по каждому режиму, по дню/неделе/всё время
- Фильтр по Ясне
- Своя позиция показывается всегда (даже если не топ)

### P4.6 Daily tournament (3 дня)
- Каждый день в 00:00 UTC новый seed для одного режима
- Все играют ровно ОДИН раз за день, лучший результат идёт в зачёт
- Top-10 в конце дня → бейджи
- Нет необходимости в реальном time-window matchmaking — это асинхронный single-player ChallengeЯ

### P4.7 Notifications (½ дня)
- «Вас обогнали в leaderboard» — браузерный push (если разрешено)
- «Новый daily challenge» — каждое утро

**Итого P4:** ~10 дней. Нужна Cloudflare аккаунт, ~$0/мес на free tier.

**Точка решения:** если за месяц после релиза 100+ DAU — катимся в P5. Иначе закрываем фичу как успешную.

## P5 — Social (1.5 недели)

**Цель:** превратить продукт из «соревнование с собой» в «соревнование с друзьями».

### P5.1 Friends list (2 дня)
- Добавляешь друга по nickname или ссылке
- Список друзей с их online-статусом
- Challenge button → отправляет приглашение
- Сторадж: KV `users:{id}:friends`

### P5.2 Challenge by URL (½ дня)
- Сгенерированная ссылка: `?duel=alice-vs-bob&mode=race-cross&yasna=суток`
- Друг кликает — попадает прямо в лобби с твоим ником

### P5.3 Replay system (3 дня)
- Каждое движение игрока пишется как event log: `{t:'click', polka:0, time:1234ms}`
- После матча — кнопка «Реплей» → проигрывание со скоростью 0.5x/1x/2x
- Сторадж: D1 `match_replays` (по requested matchId)

### P5.4 Spectator mode (2 дня)
- Перед стартом матча — checkbox «можно смотреть»
- Если включено — генерится spectator URL
- Зритель видит обоих игроков live
- Через тот же WebRTC ConnectionPool, но spectator только receives

### P5.5 Profile pages (1 день)
- `/u/alice` — публичная страница: статистика, рейтинг, последние матчи
- Можно подписаться на игрока — уведомления о его матчах

### P5.6 Clans (опционально, 2 дня)
- Группы игроков с общим тегом
- Командный leaderboard
- Только если Friends окажется востребованным

**Итого P5:** ~8 дней без кланов, +2 дня с кланами.

## P6 — Long-tail (по желанию)

Эти фичи увеличивают engagement, но не критичны.

### P6.1 Achievements (3 дня)
- Бейджи: «Первая победа», «10 матчей подряд», «Победил без ошибок 5 раз», «Все 3 креста на Ясне Суток», ...
- ~30 ачивок при старте, по одной за каждый разлом
- UI: вкладка «Достижения» в профиле, прогресс-бары

### P6.2 AI-практика (4 дня)
- Бот «Андрей» — играет на разных уровнях:
  - Easy: случайный клик каждые 3 сек, ~30% правильных
  - Medium: 50% правильных, время 12s ± 3s
  - Hard: 95% правильных, время 5s ± 1s
- Безграничная практика без живого соперника

### P6.3 Кастомизация (3 дня)
- Аватары: 50+ emoji-комбинаций
- Цвета фона комнаты (заработанные за победы)
- Анимации победы

### P6.4 Time controls (2 дня)
- Blitz (10 секунд лимит на матч)
- Classical (без лимита)
- Daily Tournament (24 часа)

### P6.5 Difficulty levels внутри режима (2 дня)
- Easy: подсказки в виде подсветки правильных позиций
- Medium: без подсказок (текущий уровень)
- Hard: Ясна без цифр и без меток (только круг)
- Insane: с вращением во время игры

### P6.6 Localization EN (3 дня)
- Английская версия UI
- Расширяет аудиторию за пределы русскоязычной

**Итого P6:** ~17 дней, по 2-4 дня на каждую фичу.

После всех фаз каталог содержит:

| Категория | Режимы | Когда добавляется |
|---|---|---|
| **Race** (3) | Опорный Крест · Управления · Веры | P0 ✅ |
| **Quiz** (1) | Антиподы | P1 |
| **Mirror** (1) | Заполни Ясну | P1 |
| **Speed** (1) | Молния (30 сек) | P1 |
| **Bidding** (1) | Аукцион ставок | P5 (опционально) |
| **Tournament** (1) | Daily challenge | P4 |
| **Practice** (1) | vs AI | P6 |

**Итого: 9 режимов × 3 Ясны = 27 уникальных матч-конфигураций**.

С добавлением новых Ясн (Зодиак, Атмосферных, Дома и т.д.) количество комбинаций растёт пропорционально — каждый новый шаблон даёт +9 матчей бесплатно.

## 4. Метрики успеха по фазам

| Фаза | Главная метрика | Целевое значение |
|---|---|---|
| **P1** | Tutorial completion rate | ≥70% |
| **P2** | Successful online matches/day | 50+ через 1 неделю |
| **P3** | Returning players (W2 retention) | ≥30% |
| **P4** | Daily tournament participation | 100+ players/day |
| **P5** | Matches played with friends (vs random) | ≥40% |
| **P6** | DAU sustained | 500+/день месяц устойчиво |

Если метрика не достигается за 3 недели после релиза фазы — НЕ катимся в следующую. Доделываем текущую или останавливаемся.

3. P5 (социал) — если есть community
4. P6 (engagement) — по желанию

## 6. Риски и не-цели

### Технические риски
- **WebRTC NAT-traversal failure 5–10%** — митигация: TURN-сервер $5/мес или Cloudflare Worker как relay (P2.3, P4)
- **Anti-cheat в JavaScript-игре невозможен полностью** — митигация: server-side timing checks + soft disqualification + community reports
- **Cloudflare free tier outage** — митигация: graceful degradation в local-only mode при недоступности сервера
- **Babel-standalone размер бандла** — митигация: при достижении 200kb переходим на pre-build через esbuild/vite

### Продуктовые риски
- **Дуэли не зайдут** — митигация: P0 уже работает как одиночная тренировка через "Сдаться". Остальное только если есть спрос.
- **Токсичность в чате/спектате** — митигация: НЕ делаем чат до P5. Только эмодзи-реакции в результате.
- **Cheating ruins competition** — митигация: P4 anti-cheat baseline + matchID + server validation. Если жалобы продолжаются — добавляем replay review.

### Не-цели (что НЕ делаем)
- ❌ Платные подписки / lootbox / NFT
- ❌ Реклама в продукте
- ❌ Voice-чат во время матча (не хочется ломать UX)
- ❌ Команды 3v3 / 4v4 — другая динамика, отдельный проект
- ❌ Stream-overlay для Twitch/YouTube — нишево
- ❌ Mobile native app — PWA достаточно

## 7. Decision gates

Перед стартом каждой фазы — checklist:

**Прежде чем делать P1:**
- [ ] P0 не имеет известных багов
- [ ] Хотя бы 1 пользователь сыграл несколько матчей
- [ ] Есть фидбек на UX лобби

**Прежде чем делать P2 (Online):**
- [ ] P1 туториал прошли ≥70% попробовавших
- [ ] Желание играть в online выражено пользователями
- [ ] Готов потратить 8 дней на инфраструктуру

**Прежде чем делать P4 (Competition):**
- [ ] 100+ матчей в неделю онлайн
- [ ] Есть бюджет на Cloudflare ($0–5/мес поначалу)
- [ ] Есть готовность модерировать читеров

**Прежде чем делать P5 (Social):**
- [ ] Активные friend-добавления (если данных нет — спросить пользователей)
- [ ] DAU стабилен и растёт

**Прежде чем делать P6 (Long-tail):**
- [ ] Все P1-P5 фичи отполированы
- [ ] DAU 500+ устойчиво

## 8. Связь с остальным продуктом

Дуэли встраиваются в общую педагогическую логику Ясны:

```
Уроки (lessons/)              ← основа знания
  ↓
Гид (tours/)                  ← закрепление в контексте каждой Ясны
  ↓
Игры (games/)                 ← активный recall
  ↓
Дуэли (games/duel/)           ← соревновательное закрепление
  ↓
Tournament                    ← мастерство
```

Каждая ступень разблокируется естественным прохождением предыдущей. Пользователь не должен играть в дуэль без понимания крестов — лобби даёт ссылки на уроки если рейтинг низкий.

## 9. Архитектурные ремарки на будущее

### Когда понадобится ESM/build pipeline
- Babel-standalone превышает 250 KB (сейчас ~70 KB)
- 30+ JS-файлов в head
- Tree-shaking стал бы экономить

Решение: переход на esbuild + один `bundle.js`. Файлы в репозитории остаются прежними, добавляется build-step. Деплой — по-прежнему статика на GitHub Pages.

### Когда понадобится TypeScript
- Когда количество файлов в `games/` превысит 20
- Когда рефакторинг `DuelTransport` ломает сразу 3+ режима

Решение: переписать `core/data.js` и `games/*` с типами, оставить `core/yasna-star.js` на JS (там JSX-heavy).

### Когда понадобится разделить prod/staging
- При первой реальной нагрузке (P4+)
- preview можно держать на главной, prod — на отдельном branch+domain
